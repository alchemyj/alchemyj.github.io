AlchemyJ modeling best practice

  1. Avoid using VLOOKUP. Use INDEX-MATCH instead.

  2. When selecting a range for lookup, avoid using the entire row/column as reference (A:C). Use A1:C100 instead.

  3. When using VLOOKUP or MATCH, sort the lookup array if possible since this speeds up the lookup.

  4. When using VLOOKUP or MATCH, fill in Match Type.

  5. Avoid using INDIRECT and OFFSET since AlchemyJ generated code cannot run in High Performance mode when they are used. They are also hard to debug, volatile, slow, single thread, do not support row/column added/deleted/moved.

  6. Try to design your worksheet so that cells can be fit into a rectangular window with minimal empty cells. This reduces the memory used during runtime.

  7. Use GetAddress for filling in address values in definition template instead of hard-coding a string value. Using GetAddress supports row added/changed/deleted since it is a function.

  8. Use GetAddress when a cell/range address string is required as input parameter of a function. This builds dependency on the reference cell.

  9. When a method input parameter is a large object or a large list of object, avoid defining Json-path for each property. Instead, get the whole JSON from input and use PopulateJsonToTable to convert the JSON string into tables. It should be faster since Json-path is a query and it is slow.

  10. Do not use array formulas except for using them for AJ Excel Extended Functions. Array formulas are slow and AlchemyJ does not fully support them.

  11. All formatting such as Date, decimal place for numbers are lost in compiled code. Do not rely on them for data formatting. Use TEXT instead.

  12. Convert dates into strings before feeding them into JSON string creation functions (MakeJson, MakeJsonFromTable, MakeJsonFromSchema) since JSON has no date data type.

  13. Use Manual Calculation mode in Excel.

  14. Use IFERROR instead of IF and ISERROR combo.

  15. You can see the parameter of a AJ Excel Extended Function by typing in the function in a formula and press CTRL+SHIFT+A.

  16. Do not use formula while a static value can be used instead.

  17. Remove unused names. Do not remove names that start with AlchemyJ_ as they are names used by AlchemyJ Studio.

  18. Avoid circular reference.

  19. Prioritize multiple-condition SUMIFS, COUNTIFS, and other IFS family function. Put the most restrictive condition first so that subsequent conditions only need to look at the smallest number of rows.

  20. When using nested IF, evaluate conditions that happen the most frequently first so other nested conditions do not need to be evaluated most of the time.
