Tutorial - testing and debugging your program

AlchemyJ provides a number of tools for users to test and debug their program efficiently.

Testing in AlchemyJ Studio (Excel)

The run function point feature lets you test the API business logic in Excel before building the project. This saves a lot of time as it can help you verify the result and make sure the logic works as expected before turning it to Java code.

For quick steps about how to run a function point in AlchemyJ Studio, you can refer to the topic of Preview your first REST API.

For the further details, you can refer to the topic of Preview your work in Excel.

Testing Java API using JUnit

AlchemyJ Compiler automatically generates a unit test project along with the main project. So makes it easier to test the generated Java API. The testing project is a JUnit project and it can be run in Excel or other Java IDE. You may also define your own test cases in JSON files.

For the details, please refer to the topic of Test Your API by Running Unit Test

Testing REST API using Build and Run

REST API package generated by AlchemyJ comes with an embedded web application server (Tomcat) which allows you to build and run the API immediately without the hassle of deploying the project to a web application server such as Websphere and Weblogic.

To build and run an API, click Build and Run under the AlchemyJ menu.

After the project is built, the API will be run automatically. It can then be tested from a web browser.

In above case, port "28082", the API name "MyAPI", Endpoint ID "SayHello" and the parameter name "name", are defined in the Endpoint Worksheet and Controller Worksheet. Therefore the following URL can call the SayHello endpoint with "Jacky" as the input:

http://localhost:28802/MyAPI/SayHello?name=Jacky

You can refer to the REST API Endpoint Worksheet and REST API Controller Worksheet for configuration details.

Please note that loading the URL using a browser submits a HTTP GET request to the endpoint only. To test other HTTP methods, you will need a web service client such as Postman and SoapUI.

From Debug Console

If you find the API does not behave as it is expected, you can build the API using Debug Mode and open Debug Console to check the API. It provides Debug Command Actions that lets to see internal state of the API such as the input it receives. You many also dump the memory into an Excel workbook so that you can check the formula and the value that causes the problem.

To learn how to turn on the debugging mode of your project, please go to Enable the Debugging Mode.

Start the Debug Console by clicking on the Launch Debug Console button on the AlchemyJ tool bar.

Take the above case as an example. We are going to overwrite the input of parameter "name" as "Tom" when the Function Point is triggered. By that, you will see the result would be the same no matter what value you have inputted. To achieve this purpose, we should have the Debugger Command Action File with follow content.

[
    {
        "action" : "set-value-to-cell",
        "parameters" : {
            "functionPoint" : "*",
            "setters" : {  
                "BizLogic!B1" : "Tom"
            }
        }
    }
]

"action" is the action name. In this case, we are using the action set-value-to-cell which sets a value to a particular cell.

"functionPoint" under "parameters" specifies which function point would trigger this debugging action; "*" means all function points would trigger the action.

"setters" specifies which cell and what value the action would write to. In this case, it sets "Tom" to cell B1 on BizLogic worksheet.
